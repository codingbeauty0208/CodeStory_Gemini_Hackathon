## API Layer

### Slide 1: RESTful API Endpoints
"The Traveller application exposes a set of RESTful API endpoints, allowing for programmatic interaction with its core resources. This is primarily defined within the `config/routes.rb` file, which utilizes Rails' `resources` helper to declare standard routes for key entities. Specifically, routes are generated for `/feedbacks`, `/slots`, `/companies`, and `/trippackages`. Each `resources` declaration automatically creates a full suite of standard RESTful actions, including `index` (to list all resources), `show` (to display a single resource), `new` (to prepare for creating a new resource), `create` (to save a new resource), `edit` (to prepare for updating a resource), `update` (to save changes to an existing resource), and `destroy` (to delete a resource). These endpoints are designed to be accessible via standard HTTP methods (GET, POST, PUT/PATCH, DELETE). Furthermore, the application is configured to support JSON API responses. This is evident through the inclusion of the `jbuilder` gem in the `Gemfile`, and the presence of `*.json.jbuilder` view files within the `app/views` directory for each resource. These Jbuilder templates define how each resource's data should be serialized into JSON format, providing clean and structured data for potential frontend applications, mobile clients, or third-party integrations."

### Slide 2: Administrative and Custom Routes
"In addition to the standard RESTful resources, Traveller incorporates specialized routes to support administrative functions and custom user interface flows. A significant component is the **Admin Panel**, which is powered by `RailsAdmin`. This powerful interface is mounted at a dedicated path: `mount RailsAdmin::Engine => '/admin', as: 'rails_admin'`. This provides administrators with a comprehensive, ready-to-use dashboard for managing all database entities—users, companies, travel packages, slots, and feedback—without requiring extensive custom development for the administrative interface itself. For **user authentication**, `Devise` handles a suite of routes generated by `devise_for :users`. These routes cover critical functionalities such as user sign-up (`new_user_registration_path`), sign-in (`new_user_session_path`), sign-out (`destroy_user_session_path`), password reset, and account editing. The application's main entry point, the root path (`/`), is explicitly configured to redirect users to the Devise sign-in page (`root 'devise/sessions#new'`), ensuring that authentication is the first step for most users. Beyond these, several **custom application routes** are defined to support specific user-facing pages or unique workflows: `home/index` for the main dashboard, `home/about` for information about the application, `mainportal/admindashboard` and `mainportal/adminpackage` for admin-specific views, and `home/myslots` for a user's personal booking overview. A custom `get '/users/sign_out' => 'devise/sessions#destroy'` route is also included for explicit sign-out handling."
