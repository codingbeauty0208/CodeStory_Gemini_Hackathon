## Core Modules

Slide 1:
"The `User` module is central to Traveller, primarily managed through the `User` model defined in `app/models/user.rb`. This model is foundational for user authentication and authorization within the application. It establishes clear relationships with other entities, as users `has_many :slots`, indicating that a single user can book multiple travel slots, and `has_many :feedbacks`, meaning users can provide multiple reviews. For robust and secure identity management, the `User` model integrates deeply with the `Devise` gem. This integration provides a comprehensive suite of authentication functionalities, including `:database_authenticatable` for password-based logins, `:registerable` for new user sign-ups, `:recoverable` for password resets, `:rememberable` for persistent sessions, and `:validatable` for email and password format enforcement. The `users` table, detailed in `db/schema.rb`, stores essential user attributes such as `email`, the `encrypted_password`, and relevant tokens like `reset_password_token`. Additionally, user profiles are enriched with `name`, `homelocation`, and `phoneno`, which are custom attributes added to the table. The `ApplicationController` extends Devise's capabilities by explicitly permitting these additional parameters (`:name`, `:homelocation`, `:phoneno`) during the sign-up process, and `:name` for subsequent account updates, ensuring comprehensive user profile management."

Slide 2:
"The travel package and company management modules are crucial for Traveller's core operations. The `Trippackage` model, defined in `app/models/trippackage.rb`, represents the individual holiday offerings available on the platform. Each `Trippackage` `has_many :slots`, indicating that a single package can be booked multiple times, and it `belongs_to :company`, signifying that each package is offered by a specific travel company. The `trippackages` table, as per `db/schema.rb`, stores a wealth of information including `package_name`, `destination`, `departure` and `arrival` dates, an allocated `budget`, a detailed `description`, `travelfrom` origin, `noofbookings` (number of bookings), and `packcountry`, along with a foreign key `company_id` linking it to the respective company. Complementing this is the `Company` model, found in `app/models/company.rb`, which manages the details of the travel agencies or providers. A `Company` `has_many :trippackages`, indicating they can offer multiple packages, and `has_many :feedbacks`, collecting all reviews related to their services. The `companies` table in `db/schema.rb` tracks key company attributes such as `companyname`, `hqlocation` (headquarters location), and a `rating`."

Slide 3:
"The `Slot` and `Feedback` modules are essential for managing bookings and capturing user sentiment, respectively. The `Slot` model, defined in `app/models/slot.rb`, represents a confirmed booking by a user for a specific travel package. It establishes two critical `belongs_to` associations: `belongs_to :user`, linking each booking to the individual who made it, and `belongs_to :trippackage`, connecting it to the specific travel package being booked. The primary attribute stored in the `slots` table (from `db/schema.rb`) is `bookingtime`, capturing the precise time of the booking. The `Feedback` model, located in `app/models/feedback.rb`, is designed to collect user reviews. Like `Slot`, it also has `belongs_to :user` and `belongs_to :company` associations, ensuring that each feedback entry is tied back to both the user who provided it and the specific company whose service is being reviewed. The `feedbacks` table in `db/schema.rb` stores the numerical `rate` and a textual `descr` (description) of the feedback. Both `SlotsController` and `FeedbacksController` implement standard CRUD (Create, Read, Update, Delete) operations. These controllers include `before_action` callbacks to `set_slot` or `set_feedback` for specific actions, and for `SlotsController` and `FeedbacksController`, `set_current_trippackage` and `set_current_company` respectively, retrieve related data, often utilizing session variables to maintain context for creating new records related to existing packages or companies. For example, when booking a package or leaving feedback, the system intelligently associates the new record with the `current_user` and the relevant `Trippackage` or `Company`."
