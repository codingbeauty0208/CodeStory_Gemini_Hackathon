## Data Layer

Slide 1:
"The data layer of the Traveller application is designed for robustness and efficiency, primarily utilizing a relational database system. The core database technology employed is **MySQL**, as explicitly identified by the `mysql2` gem present in our `Gemfile` and confirmed by the adapter configuration within `config/database.yml`. This choice signifies a preference for a widely supported, high-performance RDBMS. The `config/database.yml` file meticulously defines the connection parameters for distinct environments: `development`, `test`, and `production`. For the `development` environment, it specifies a default username of 'root' and a hardcoded password 'rajshr!2934ashvar' (though this is not best practice for production). In `production`, the password is dynamically fetched from an environment variable, `ENV["TRAVELLER_DATABASE_PASSWORD"]`, enhancing security. All three environments typically connect to a `host` of 'localhost'. The application leverages **ActiveRecord**, Ruby on Rails' powerful Object-Relational Mapping (ORM) framework. Every model within the application, including the base `ApplicationRecord` and specific models like `User`, `Trippackage`, `Company`, `Slot`, and `Feedback`, inherits from `ActiveRecord::Base`. This allows developers to interact with database records using Ruby objects and methods, abstracting away raw SQL queries and simplifying data manipulation and persistence logic."

Slide 2:
"The database schema, meticulously defined in `db/schema.rb`, outlines the structure of our persistent data and the relationships between different entities. The `users` table is fundamental, storing critical information for user authentication and profiles. It includes standard Devise fields like `email` and `encrypted_password`, along with `reset_password_token` for password recovery, and `remember_created_at` for session management. Additionally, custom attributes such as `name`, `homelocation`, and `phoneno` are included to enrich user profiles. The `companies` table is designed to hold information about the travel service providers, containing `companyname`, `hqlocation`, and a numerical `rating`. This allows for tracking and organizing different partners. For the actual travel offerings, the `trippackages` table stores detailed package information: `package_name`, `destination`, `departure` and `arrival` timestamps, a `budget` (integer), `description`, `travelfrom` location, `noofbookings` (number of bookings), `packcountry`, and importantly, a `company_id` foreign key. This `company_id` links each package directly to the `companies` table, establishing a `belongs_to` relationship in the `Trippackage` model and `has_many` in the `Company` model. Bookings are managed in the `slots` table, which records `bookingtime` and crucially includes `user_id` and `trippackage_id` foreign keys. These foreign keys establish `belongs_to` relationships, linking each slot to a specific `user` and `trippackage`, allowing for accurate tracking of who booked what. Finally, the `feedbacks` table stores user reviews, comprising a `rate` (integer) and a textual `descr`. It also includes `user_id` and `company_id` foreign keys, enabling the system to associate feedback directly with the user who provided it and the company that received it, reinforcing the `belongs_to` and `has_many` associations in the respective models."
