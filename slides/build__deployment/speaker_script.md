## Build & Deployment

### Slide 1: Frontend Asset Compilation
"The Traveller application employs a modern and efficient workflow for handling its frontend assets, ensuring optimized delivery and ease of development. For **JavaScript bundling**, the `package.json` file dictates the use of `esbuild`. The `build` script defined (`esbuild app/javascript/*.* --bundle --sourcemap --outdir=app/assets/builds --public-path=assets`) compiles all JavaScript entry points from `app/javascript` into a single bundle, outputting it to `app/assets/builds`. This process includes source maps for easier debugging in development. Complementing this, **CSS pre-processing** is managed by `sass`, also configured in `package.json`. The `build:css` script (`sass ./app/assets/stylesheets/application.bootstrap.scss:./app/assets/builds/application.css --no-source-map --load-path=node_modules`) compiles our Bootstrap-integrated SCSS files into a minified CSS bundle. Both `jsbundling-rails` and `cssbundling-rails` gems seamlessly integrate these external JavaScript and CSS build tools directly into the Rails asset pipeline. This means that during development, changes are live-reloaded, and in production, assets are precompiled and fingerprinted for efficient caching. The `app/assets/config/manifest.js` file plays a crucial role by explicitly linking these asset directories (`../images`, `../builds`) and specific stylesheets (`application.js`, `login.css`, `about.css`), ensuring all necessary frontend resources are correctly identified and processed by the Rails asset pipeline."

### Slide 2: Development and Production Environment Configuration
"The application's environment configuration is finely tuned for different stages of its lifecycle: development, testing, and production. In the **development environment**, defined in `config/environments/development.rb`, `config.cache_classes = false` is set to enable instant code reloading upon changes, which is ideal for rapid iteration. `config.eager_load = false` prevents loading the entire application on boot, contributing to faster startup times. For file uploads, `config.active_storage.service = :local` uses the local disk for storage. `config/puma.rb` also configures a `worker_timeout` of `3600` seconds specifically for development, allowing longer debugging sessions. The **test environment** (`config/environments/test.rb`) uses `config.cache_classes = true` for consistent and isolated test runs. `config.eager_load` is conditionally enabled based on the `CI` environment variable, suitable for continuous integration pipelines. `config.active_storage.service = :test` directs file uploads to a temporary local directory, ensuring a clean state for each test. Furthermore, `config.action_mailer.delivery_method = :test` ensures emails sent during tests are captured in an array rather than being sent to actual recipients. For the **production environment** (`config/environments/production.rb`), performance and reliability are paramount. `config.cache_classes = true` and `config.eager_load = true` are enabled for maximum performance. `config.assets.compile = false` signifies that assets are expected to be precompiled during deployment, typically by an external web server like NGINX, rather than on-demand by Rails. While `config.active_storage.service` defaults to `:local`, commented-out options for Amazon S3, Google Cloud Storage, and Microsoft Azure suggest readiness for cloud-based file storage. Production logging is set to `info` level, with `request_id` tagging, and `config.cache_store` is set to `:null_store` by default, but is ready for a production-grade cache store like memcached. Finally, sensitive configurations are managed via `config/credentials.yml.enc`, an encrypted file that secures API keys and other secrets, with access controlled by a `master.key`."

### Slide 3: Local Deployment and Setup Scripts
"To streamline the developer workflow and ensure consistent environment setup, Traveller includes several crucial scripts and configurations. The `Procfile.dev` file acts as a local deployment manifest, defining the concurrent processes necessary for a fully functional development environment. It outlines three key services: `web` for the Rails application server (using `bin/rails server`), `js` for continuous JavaScript compilation via `yarn build --watch`, and `css` for continuous CSS compilation with `yarn build:css --watch`. This setup, leveraging tools like `esbuild` and `sass`, ensures that changes to both backend and frontend assets are immediately reflected during development. The `bin/dev` script simplifies initiating this multi-process environment. It first checks for and installs `foreman`, a process management tool, and then executes `foreman start -f Procfile.dev "$@"`, thereby launching all specified services simultaneously. This provides a single command to get the entire development stack up and running. For initial project setup and ongoing updates, the `bin/setup` script is provided. This idempotent Ruby script handles essential tasks such as installing all Ruby gems using `bundle install`, preparing the database by running `bin/rails db:prepare` (which includes migrations and seeding if applicable), clearing old log files and temporary directories (`bin/rails log:clear tmp:clear`), and finally restarting the application server to apply all changes. These scripts collectively standardize the development setup, making it efficient for new contributors and maintaining consistency across different development machines. The Puma web server, configured in `config/puma.rb`, further specifies operational parameters such as thread counts (dynamically set by `RAILS_MAX_THREADS` and `RAILS_MIN_THREADS` environment variables), the listening `port` (defaulting to 3000), and the `pidfile` path, all critical for controlled application execution."
